## R1 minor comment 6 / R1-6 (major) — upstream pipeline versions & citations
The manuscript methods name FastQC, Trimmomatic, STAR, featureCounts, and DESeq2 but
do not give version numbers.

**RESOLVED for DESeq2 — decision: adopt the re-analysis version as the
stated version, do not chase the original.** DESeq2 is a downstream
statistical step we can and did run ourselves directly on the deposited
count matrices; every DE/GSEA number in this revision's supplementary
material is the literal output of **DESeq2 v1.38.3**
(`logs/09_sessionInfo.txt`). Rather than trying to forensically recover
whatever version the original submission used (unrecoverable — see below),
the simplest, honest, and fully defensible path is for the revised
manuscript's Methods to state **"DESeq2 v1.38.3"** as the version used for
this (revised) analysis — every number in the revision is truly generated by
that exact version, so this is a factual statement, not a guess. This
closes out the DESeq2 line item. Do not apply the same logic to the
upstream tools below.

- [ ] FastQC version used for original QC
- [ ] Trimmomatic version + exact trimming parameters used
- [ ] STAR version + exact alignment parameters/index build (annotation
      source/release) used
- [ ] featureCounts version + exact parameters (strandedness setting, etc.)
- [x] ~~Original DESeq2 version~~ — resolved above; report v1.38.3 as the
      version used for the revised analysis.
- [ ] Citations for Trimmomatic, STAR, featureCounts, DESeq2 (standard citations,
      but need author confirmation of which papers they intend to cite)

**Why "just use the same version" does NOT transfer to FastQC/Trimmomatic/
STAR/featureCounts, even though it does for DESeq2:** those four tools
*produced* the deposited count matrices themselves (FASTQ -> BAM -> counts).
We never touched raw FASTQ files — we only ever analyzed the
already-deposited, already-fixed counts. The counts on GEO are the fossil
record of whatever specific STAR/Trimmomatic/featureCounts run actually
happened; we cannot substitute a different version's name into the Methods
without it being potentially false, since we have no way to verify it
matches what actually produced those specific numbers. (Candidate version
numbers found circumstantially on this shared server are recorded below for
reference, but are explicitly NOT proposed as a substitute for author
confirmation.)

## R1-6 / R1 minor comment 7 — original GSEA tool identity
The manuscript says "GSEA algorithm with permutation-based significance estimation"
without naming software. This re-analysis used **fgsea 1.24.0** (preranked,
multilevel estimation), which is almost certainly NOT the exact tool/settings used
to generate the original Fig. 6B/6D NES values, given the sign discrepancy found for
the podocyte gene set (see README_analysis_log.md, "Key findings" #1). The authors
should state definitively what was originally used (Broad GSEA desktop app,
clusterProfiler, fgsea, GSEApy, etc.) and, ideally, share the original script so the
sign discrepancy can be root-caused rather than left as an open question.
- [ ] Name of original GSEA software/package + version
- [ ] Original permutation count / seed (if any) used
- [ ] Original exact contrast direction/orientation used for ranking (to help
      resolve the sign discrepancy noted in README_analysis_log.md)

## R2 major comment 4 — original A1 QC criteria
The manuscript states A1 (A-ADR1) was excluded for "reduced glomerular purity" but
does not give the quantitative QC criterion/threshold used. This re-analysis
independently corroborated low glomerular purity for A-ADR1 using a post-hoc
podocyte:tubular marker CPM ratio (`logs/00_qc_report.txt`), but this is a
re-derivation for this response letter, not a recovery of the original stated
criterion. The authors should state their original quantitative threshold (if one
was pre-specified) or explicitly adopt the marker-ratio approach shown here as the
documented criterion going forward.
